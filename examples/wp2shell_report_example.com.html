Analyst notes

• Detected version is exposed to the full wp2shell pre-auth RCE chain (CVE-2026-63030 + CVE-2026-60137). Treat any internet-facing instance that ran this version as potentially compromised — patching closes the route but does not remove a backdoor planted beforehand.

Remediation

  1. Take a verified backup first.

    Back up the database and the full file tree before making changes, so you can restore and later forensically compare.

  2. Update WordPress Core.

    Move to a fixed release — 6.8.6 · 6.9.4 · 6.9.5 · 7.0.2 or later on the matching branch. Then confirm with wp core version or Dashboard → Updates.

  3. Confirm the site still works.

    Verify the public site and admin load correctly after the update; check for plugin/theme conflicts.

  4. Assume compromise if it was exposed while vulnerable.

    The patch closes the route but does not remove a backdoor planted beforehand. Inspect themes, plugins, uploads/, wp-config.php, scheduled tasks (wp-cron), and the admin user list for persistence, and run file-integrity checks against known-good core.

  5. Add defence-in-depth at the edge.

    Where immediate patching isn't possible, deploy WAF rules to block malicious REST batch requests and SQL-injection patterns, restrict access to /wp-json/batch/v1, and monitor REST API logs. Managed WAFs shipped rules for this chain on 17 Jul 2026.

  6. Rotate secrets if compromise is suspected.

    Regenerate the wp-config.php auth salts, reset admin and database passwords, and rotate any API keys stored on the host.

  7. Re-scan to confirm.

    Run this scanner again; the verdict should read Patched and the version should sit at or above the fixed release.
